# AutoResearch 调研 v2 —— 按"应用算法团队的工作流"重组

> 调研日期：2026-06-24 ｜ 团队画像：自驾/机器人**应用算法团队**，在社区基线上做改进。
> 真实诉求：**看到一篇 paper / 一个 product 的好想法 → 提炼核心 idea → 复现确认有效 → 融进我们现有的框架/方案 → 用实验验证效果**。

---

## 0. 重新定位：你们要的不是"一个项目"，是一条流水线

上一版我按"研究社区怎么命名这些项目"切成了 5 类，对你们太细。**真正的视角**应该是：你们的工作流是一条循环，而 2026 年社区（尤其 3 月 Karpathy 发 AutoResearch 之后）已经把它拆成**几段可组合的能力**——每段有对应的工具，关键是它们**叠在一起用**，不是二选一。

```
        ┌─────────────────────────────────────────────────────────┐
        │  ① 发现&提炼          ② 复现确认         ③ 融入现有框架      │
        │  (paper/product   →  (跑通、确认    →  (改我们自己的       │
        │   的 idea 提炼)        确实有效)          大代码库)         │
        │        ↑                                      │           │
        │        │            ④ 实验验证效果（贯穿②③）    │           │
        │        └──────────  (改→跑→看指标→保留/回滚) ←──┘           │
        └─────────────────────────────────────────────────────────┘
```

**一句话**：你们需要的是「**代码 agent（能读懂你们框架）+ 自动实验循环（验证想法）+ 复现/提炼前端**」三者拼成的一套东西。下面按这三段 + 一个最关键的工程前提来组织。

---

## 1. 核心引擎：想法 → 自动实验验证 ★最匹配"帮我做实验，验证想法效果"

这一段是你们诉求的正中心。范式定义者就是 Karpathy 3 月发的 AutoResearch。

### 1.1 Karpathy's AutoResearch ⭐⭐ 必看（范式定义者）
- **是什么**：2026-03-07 发布，早期就 ~66k stars / 9.6k forks。让一个 AI 编码 agent **整晚自动跑 ML 实验**：你在一个 markdown 文件（`program.md`）里写清"想改进什么 + 怎么判断更好 + 什么不能动 + 什么时候停"，把 agent 指向 repo，走人；早上回来得到一串被验证过的改进 + 所有尝试的日志。Karpathy 自己形容它"part code, part sci-fi"。
- **机制（"棘轮 ratchet"）**：propose（改任意代码）→ 训练固定 5 分钟 → 看指标 → 改进则 `git commit` 成新基线，没改进则 `git reset` 丢弃。约 12 实验/小时，一晚 80–100 个。固定时间窗让**所有实验直接可比**，且单次失败只损失 5 分钟。
- **关键区别**：这**不是超参搜索**（Optuna/Ray Tune 是在预定义空间里搜）——agent 可以改架构、优化器、tokenization 等**任意代码**，搜索空间是开放的。
- **底座**：跑在 nanochat 上，单文件 `train.py` ~630 行、单 GPU、单指标 `val_bpb`（与词表无关，所以能公平比较改了架构/分词的版本）。**它本身不带 agent**——"在 repo 里开你的 Claude Code / Codex"即可（→ 见第 2 节，这就是组合点）。MIT 许可。
- **真实战绩**：Karpathy 在 8×H100 上两天跑 ~700 实验，叠出约 20 项改进，把 "Time to GPT-2" 从 2.02h 降到 1.80h；Shopify CEO Tobias Lütke 用它在**自己的私有数据**上 37 个实验拿到 19% 提升——**证明能迁移到非公开/专有场景**（对你们很关键）。
- **对你们的意义**：这正是"我有个想法，帮我跑实验看有没有用"的最小可用范式。把 `program.md` 写成"在我们的 planner/感知模块里尝试 XX 这个 paper 的想法"，指标换成你们的仿真分数，就能套。**但前提是指标要满足第 5 节的条件。**
- 链接：`github.com/karpathy/autoresearch`（搜 autoresearch）；导读见 DataCamp / Verdent guide。

### 1.2 Curie ⭐ 学术严谨版（最贴合你们"复现→扩展→融入"）
- **是什么**：密歇根大学 + Cisco 的开源框架（`Just-Curieous/Curie`），目标是**严谨的自动化实验**。结构 = Architect Agent（定假设、设计实验、反思）+ 多个 Technician Agent（实现、执行）+ 中间的 **Experimental Rigor Engine**（保证可靠性/受控/可复现）。
- **为什么对你们对口**：它的官方案例就是对一个已有方法做 **复现（Reproduce）→ 扩展（Extend）→ 挑战（Challenge）**——几乎是你们工作流的标准模板。它明确支持"从论文里提炼高层贡献"**或**"从开源项目里提炼核心系统行为"作为起点。
- **配套**：EXP-Bench（"AI 能不能做 AI 研究实验"，46 个来自真实论文/开源项目的任务）；还有针对 MLE-bench 找最优 ML 方案的 AutoML 功能。比最强基线在"正确回答实验问题"上有 3.4×。
- 链接：arXiv 2502.16069 ｜ `github.com/Just-Curieous/Curie`

### 1.3 组件级精修：AIDE / MLE-STAR（验证"想法的哪个部件真有用"）
- **AIDE**（Weco AI）：对解法做**树搜索**，是 MLE-bench 上最强的 scaffold。想自建实验 agent，AIDE 是成熟骨架。
- **MLE-STAR**（Google，arXiv 2506.15692）：先用搜索引擎拉来 SOTA 模型形成初始方案，再用**消融实验（ablation）引导**对特定流水线部件做定向精修，最后做集成。在 MLE-bench Lite 上拿牌率 64%（最强基线 25.8%）。**"消融引导的定向精修"**这个思路，正好用来回答"我融进来的这个想法，到底是哪一块带来了提升"。

> **这一段的硬约束**：AutoResearch / 棘轮式循环要能自动跑，需要同时满足——**① 范围可控（不要让它在整个框架里乱改）② 一个确定性的标量指标 ③ 固定的单次实验时间预算**。缺一个，自动循环就会崩（范围太大压不住、成功与否判不清、或实验之间不可比）。这对自驾/机器人是真问题，见第 5 节。

## 2. 落地基座：能在你们现有框架上动手的代码 Agent

你们最核心的隐含约束是"**要改我们自己的大代码库**"，而不是白纸生成一个新 repo。这正是 paper2code 类工具的短板，要靠**仓库级代码 agent**补。第 1 节的实验循环就是**跑在这一层之上**的。

| 工具 | 形态 | 对你们的点 |
|---|---|---|
| **Claude Code / OpenAI Codex** | CLI / 终端 | 直接在你们 repo 里跑，能读/搜代码、跑测试、改多文件。Karpathy 的 AutoResearch 默认就配它俩。 |
| **Devin** | 云端自治 SWE | 当成"可以派活的工程师"，适合后台跑较大改造任务、并行多实例。 |
| **Cursor / Windsurf** | Agentic IDE | Windsurf 的 codemaps 给大代码库建结构表示，便于在大 repo 里推理依赖/架构——对你们这种"一整套方案"的代码量友好。 |
| **OpenHands** | 开源仓库 agent | 想自建/可控、要开源可改的首选基座。 |
| **SWE-agent** | 仓库级 issue 解决 | 学术界仓库级编码 agent 的代表，研究"怎么让 agent 在已有 repo 里干活"。 |

**通用范式**（2026 主流）：plan → execute → validate 循环，agent 主动探索代码库、用 git worktree 隔离改动、最后人审（"intern model"）。**对你们的含义**：把"融入现有框架"这步交给代码 agent 做成可审查的 PR，再用第 1 节的循环去验证效果——分工清晰。

---

## 3. 流水线前端：从 paper / product 提炼想法 + 复现确认

这一段对应你们的 ①②（提炼 + 复现确认确实有效）。

- **提炼想法**：Deep Research（系统综述 + 找到原始实现/相关工作），把一篇 paper / 一个 product 的核心机制扒清楚。
- **复现确认**（已在 v1 详述，这里只点对口性）：
  - **Paper2Code / PaperCoder**（arXiv 2504.17192）：论文 → 可跑代码仓库，o3-mini 跑一篇 ~$0.5–0.7。适合"先独立复现出一版，确认 idea 成立"。
  - **AutoReproduce**（arXiv 2505.20662）：亮点是 **paper lineage**——顺着引用关系从被引文献挖"论文没写明的实现细节"，并同步生成单元测试。**"细节对不上、隐式知识缺失"正是复现自驾/感知算法最痛的点**，这思路值得借鉴甚至自研一套。
- **注意**：这两个是"白纸生成新 repo"，**不擅长直接改你们的框架**。正确用法是：用它们独立复现/确认 → 确认有效后，再用第 2 节的代码 agent 把想法**移植**进你们的框架。

---

## 4. 端到端"AI 科学家"——可以借鉴，但别直接套

AI Scientist v2 / AI-Researcher / Agent Laboratory 这类"idea→实验→论文"全自动系统，对你们**不是直接可用的产品**（你们不需要它写论文），但有两点值得借鉴：
- **AI-Researcher**（`HKUDS/AI-Researcher`，NeurIPS 2025 spotlight，有生产版 novix.science）：它的 **Validation Phase + Refinement Phase** 把"实现→系统测试→收集指标→迭代优化"做成了清晰闭环，这套**验证/精修循环的工程设计**可以照搬到你们的内部工具里。
- **AI Scientist v2**（`SakanaAI/AI-Scientist-v2`）：实验管理 agent + 树搜索的工程参考。

## 5. ★ 对自驾/机器人最关键的一点：瓶颈是 eval，不是 agent

这是整份调研里我最想强调的、也是最可落地的判断。

- **为什么 Karpathy 的循环能成立**：因为 `val_bpb` 这个指标**快（5 分钟跑完）、确定性、可比、是单一标量**。当实验比人快 100 倍时，**评测管线本身就变成了瓶颈**（社区共识）。
- **为什么你们不能直接套**：你们"验证想法效果"的真实指标是**闭环仿真 + 多目标**（安全/碰撞、舒适、通行时间、规则合规…），它**慢、随机（多 seed）、多维、还容易被钻空子**。
- **所以你们要先投入的，不是接哪个 agent，而是把评测搭对**：
  1. **一套有界的场景套件**（固定、可复现的若干 driving/manipulation 场景），让每次实验**可比**。
  2. **把多目标压成一个可比的标量**（加权或带约束的主指标），否则棘轮"keep/revert"判不了。
  3. **固定单次实验预算**（比如每个想法跑固定数量场景 × 固定 seed），控制爆炸半径。
  4. **Goodhart 防护**：自动循环会**无情地钻指标空子**。如果你们的安全指标可被"投机取巧"地刷高，自动 agent 一定会找到——所以安全/合规这类指标要做成**难以 game 的、最好带硬约束/否决项**。
- **一句话**：**先有"快速、确定性、可标量化、抗 game"的仿真评测，AutoResearch 式自动验证才谈得上落地。** 这套评测本身就是你们最值得自研的核心资产。

## 6. 推荐落地方案：搭一套，而不是选一个

**最小可用组合（建议起点）**：
```
Claude Code / Codex        ← 读懂并改动你们现有框架（落地基座）
   └─ 套一层 AutoResearch 式棘轮循环（program.md 写"试某 paper 的想法 + 约束 + 停止条件"）
        └─ 指标 = 你们自研的「有界场景套件 + 标量化分数」（第 5 节，前提）
前端：Deep Research 提炼 idea + Paper2Code/AutoReproduce 独立复现确认
```

**分阶段推进**：
1. **先搭评测**（第 5 节）——这是地基，没它后面都空转。
2. **跑通单点循环**：选一个**范围小、指标清晰**的子模块（如某个 planner cost 项、某个 loss/正则、某个数据增强），用 Claude Code + AutoResearch 棘轮验证一个想法。
3. **接前端**：把"读 paper → 提炼 → 复现确认"用 Deep Research + Paper2Code/AutoReproduce 串起来。
4. **严谨化**：需要"复现→扩展→挑战"式的严谨实验时，引入 **Curie**；需要"想法里哪块有用"的归因时，用 **MLE-STAR 的消融引导**思路。

**选型速查**：

| 你要干的事 | 用什么 |
|---|---|
| 让 agent 在我们框架里改代码、出可审 PR | **Claude Code / Codex**（可控）、Devin（云端大任务）、OpenHands（开源自建） |
| 有个想法，帮我跑实验看有没有效果 | **Karpathy AutoResearch 范式**（套在上面的代码 agent 上） |
| 严谨地"复现→扩展→挑战"一个方法/开源项目 | **Curie** |
| 判断融进来的想法是哪一块带来提升 | **MLE-STAR**（消融引导精修）、AIDE |
| 独立复现一篇 paper 确认 idea 成立 | **Paper2Code**；细节难复现时 **AutoReproduce** |
| 借鉴"验证/精修闭环"的工程设计 | **AI-Researcher** 的 Validation/Refinement Phase |

## 7. 会踩的坑（自动循环特有）

- **棘轮 = 局部搜索**：只接受"立刻变好"的改动，**做不到"先变差再变好"**（GitHub issue #22）。所以**大的架构级改造**别指望它自己搜出来——那种要人先给方向/先做出半成品，再让循环精修。
- **Goodhart / 指标作弊**：见第 5 节，自动 agent 会钻一切可钻的指标空子。
- **RLHF 保守性**：Karpathy 观察到 agent 在开放性问题上"cagy and scared"（偏保守、不敢大胆试）——因为 RLHF 奖励安全输出。开放探索类任务效果会打折。
- **多目标 ≠ 单标量**：自驾的安全/舒适/效率本质多目标，硬压成一个分数会丢信息；可考虑"主指标 + 一票否决约束"的折中。
- **复现的隐式知识**：论文常省略关键训练细节；AutoReproduce 的 paper lineage、"拿原作者 repo 当 ground truth"是主要应对手段。
